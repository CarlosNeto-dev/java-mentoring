Hierarquia de Classes do Java 📈

Neste capítulo, entramos na fase de arquitetura avançada. Vamos aprender como o Java permite que classes se relacionem, compartilhem código e se comportem de formas diferentes. É aqui que você para de criar classes isoladas e começa a construir ecossistemas de objetos.

Nossa jornada será dividida nestes três tópicos principais 🚀:

  • 🧬 4.1 Herança e Polimorfismo: O coração da reutilização. Entenda como uma classe "filha" pode herdar tudo de uma classe "pai" e, ao mesmo tempo, como um único objeto pode assumir múltiplas formas dependendo do contexto.
  • 🏗️ 4.2 Classes Abstratas e Interfaces: Os contratos do Java. Aprenda a criar modelos que definem o que deve ser feito, deixando para as classes específicas a liberdade de decidir como executar a tarefa. É a base dos grandes frameworks.
  • 📦 4.3 Pacotes e Organização de Código: O toque final do profissional. Aprenda a arrumar a casa, separando seu código em pastas lógicas (packages) para evitar conflitos e tornar seu projeto fácil de navegar e manter.

Herança e Polimorfismo 🧬🎭

A Herança (Inheritance) é o mecanismo que permite que uma classe herde atributos e métodos de outra. Já o Polimorfismo é a capacidade de um objeto ser tratado como sua classe pai, mas ainda assim agir de sua própria maneira única.

Por que isso é importante?

  • Reutilização: Você não precisa reescrever o código de "vida" ou "nome" para cada tipo de personagem.
  • Flexibilidade: Você pode criar uma lista de "Characters" e colocar Guerreiros, Magos e Arqueiros juntos. O Java saberá como cada um deve atacar.

💻 Construindo a Árvore de Família

No exemplo abaixo, veja como a classe Warrior estende Character (Herança) e como o método action() se comporta de forma diferente, sobrescrevendo a ação padrão (Polimorfismo). 


// Classe Parente (Superclasse)
public class Character {
    protected String name; // protected permite que os filhos acessem

    public Character(String name) {
        this.name = name;
    }

    public void action() {
        System.out.println(name + " realiza uma ação genérica.");
    }
}

// Classe Filha (Subclasse usando 'extends')
public class Warrior extends Character {

    public Warrior(String name) {
        super(name); // Repassa o nome para o construtor do Pai
    }

    // Polimorfismo: Sobrescrevendo o método do Pai
    @Override
    public void action() {
        System.out.println(name + " ataca ferozmente com sua espada! ⚔️");
    }
}

// No Main:
public class Main {
    public static void main(String[] args) {
        // Polimorfismo na prática: a variável é Character, mas o objeto é Warrior
        Character hero = new Warrior("Carlos");

        hero.action(); // Saída: Carlos ataca ferozmente com sua espada! ⚔️
    }
}

Classes Abstratas e Interfaces 🏗️

Em projetos grandes, às vezes precisamos forçar regras. Queremos garantir que toda classe que representar um "Inimigo" tenha, obrigatoriamente, um método de atacar. Para isso usamos as Classes Abstratas e as Interfaces.

Classes Abstratas ☁️

São classes que servem apenas como molde. Você não pode usar o `new` nelas. Elas podem ter código pronto para os filhos herdarem, mas também podem ter "métodos abstratos" (métodos sem corpo) que os filhos são obrigados a preencher.

Interfaces 📜

A Interface é um contrato 100% rígido. Nela, você só diz "o que" o objeto deve fazer, sem dizer "como". Uma classe pode assinar esse contrato usando a palavra implements. 


// A Interface (O Contrato)
public interface Playable {
    void pressStart(); // Quem implementar, é OBRIGADO a ter esse método
}

// A Classe assinando o contrato
public class Game implements Playable {
    private String title = "Java Quest";

    @Override
    public void pressStart() {
        System.out.println("Iniciando o jogo: " + title + " 🎮");
    }
}

Pacotes e Organização de Código 📁

Quando seu projeto passa de 5 ou 10 classes, deixá-las todas na mesma pasta vira um caos. No Java, nós separamos nosso código em Packages (Pacotes), que nada mais são do que pastas no seu computador.

  • Package: A primeira linha de qualquer classe Java diz em qual pasta ela mora.
  • Import: Se você precisa usar uma classe que está em outra pasta, você tem que "importá-la" para o seu arquivo atual.
Padrão de Mercado: As empresas geralmente nomeiam pacotes com o domínio do site ao contrário. Exemplo: br.com.monitoria.models. Isso garante que nunca haverá conflito de nomes, mesmo se você juntar o seu código com o de outra empresa! 🌐 

// ARQUIVO: src/com/monitoria/models/Student.java
package com.monitoria.models; // Declarando onde a classe mora

public class Student {
    public String name = "Carlos";
}

// --------------------------------------------------------

// ARQUIVO: src/com/monitoria/main/System.java
package com.monitoria.main; // Mora em outra pasta

import com.monitoria.models.Student; // Trazendo a classe para cá!

public class System {
    public static void main(String[] args) {
        Student s1 = new Student();
    }
}
